# Project Proposal: StudyRAG — AI Document Assistant

## 1. Statement of Purpose
**Main Goal**: To create a private, locally-hosted AI study assistant that allows students to interact with their PDF textbooks and notes using Retrieval Augmented Generation (RAG).

**Problem Solved**: Students often struggle to find specific information in long PDFs. Traditional LLMs have context window limits or privacy concerns. This project provides a local, secure solution that "knows" the student's specific materials.

**Importance**: It enhances study efficiency by providing instant, cited answers from personal documents without sending data to third-party cloud AI providers.

---

## 2. Data to Be Used for Training
**Dataset**: User-uploaded PDF documents (textbooks, lecture slides, research papers).
**Key Features**: Unstructured text data. The system handles multiple records (PDF chunks).
**Preprocessing**: 
- **PDF Extraction**: Using `pypdf` to convert pages to raw text.
- **Chunking**: Text is split into 500-character segments with 100-character overlap to preserve context across boundaries.
- **Cleaning**: Removal of excessive whitespace and non-text elements to optimize embedding quality.

---

## 3. AI Tools or Packages to Be Used
**Programming Language**: Python 3.13
**Frameworks/Libraries**:
- **FastAPI**: Web backend and API routing.
- **Ollama**: Local inference engine for Embeddings (`nomic-embed-text`) and Chat (`llama3.2`).
- **NumPy**: Vector math and in-memory cosine similarity calculations.
- **PyPDF**: PDF text extraction.
**Supporting Tools**:
- **Antigravity IDE**: Development and testing.
- **Uvicorn**: ASGI server for deployment.
- **Vanilla JS/CSS**: Frontend interface.

---

## 4. Process / Workflow Overview
1.  **Data Collection**: User uploads PDF via the web UI.
2.  **Preprocessing**: System parses text and splits it into logical chunks.
3.  **Embedding**: Each chunk is converted into a 768-dimensional vector (embedding) using the `nomic-embed-text` model in Ollama.
4.  **Vector Storage**: Chunks and vectors are stored in an in-memory `VectorStore`.
5.  **Zero-Shot Retrieval**: When a student asks a question, the question is embedded, and the system performs a semantic search. No active training phase is required for new documents.
6.  **Grounded Generation**: The top-5 most relevant text chunks are sent to the LLM (Llama 3.2) as mandatory context.
7.  **Output**: The AI generates a response grounded *only* in that context, which prevents "hallucinations" and ensures factual accuracy.

---

## 5. Expected Output
**System Output**: Markdown-formatted answers generated by the AI, explicitly citing the source document and chunk index.
**Success Measurement**:
- **Relevance**: How well the answer matches the provided context.
- **Latency**: Speed of retrieval and generation (target < 3 seconds).
- **Usability**: Ease of document management in the sidebar.

---

## 6. Final Envisioned System of Deployment
**Delivery**: Web Application.
**Hosting**: Local hosting (localhost:8000) with local AI inference (Ollama).
**Portability**: The system is designed to be containerized using Docker for easy deployment on any workstation with an GPU/NPU.

---

## 7. Prototype (Minimum Viable Product - MVP)
**Simplest Version**: A single-page web app where a user can upload one PDF and ask questions about it.
**Functionality**:
- PDF Upload and parsing.
- Real-time chat with AI.
- Context-aware responses.
**Testing**: Demonstrated by uploading a technical PDF (like `test_rag.pdf`) and verifying the AI can correctly summarize it or find specific facts within it.
